# 광명 스마트데이터포털 — 데이터 교체 상세기획서

> 작성 기준일: 2026-06-23 · 범위: 전체 화면 · 원칙: 디자인/레이아웃/인터랙션 불변, **정적 시드 데이터만 교체**, 실명(실제 기업·시스템명) 사용

## ✅ 확정 결정사항 (2026-06-23)
1. **데이터 마일 = 데이터플랫폼으로 재정의** — CCM/VCM 탄소거래 표현 제거, 정책플랫폼 84종 연계·데이터 개방·API 호출 중심으로 교체
2. **문서 반영 = 전체 문서** — 세부데이터선별·기능정의서·프로세스정의서·아키텍처(정의서/설계서)까지 검토해 필드·주기·단위 정밀 반영
3. **관리자 운영데이터 = 현실화** — 데이터목록·플랫폼 관리는 실제 데이터셋/플랫폼과 연동되게 보정, 공지·자료·팝업·계정은 현실적 톤으로
4. **착수 순서 = Phase 1(`src/data.ts`)부터**

## 1. 기본 원칙 / 데이터 성격 구분

교체 대상 데이터는 두 종류로 나뉘며, 처리 방식이 다르다.

| 구분 | 정의 | 출처 | 처리 |
|---|---|---|---|
| **(A) 확정 데이터** | 시스템명·사업자명·데이터셋명·필드명·단위·분류체계 | 기획 문서가 **정답** | 문서값 그대로 반영 |
| **(B) 수치 시드** | 발전량·이용자수·수위 등 실측 "값" | 문서에 **라이브 값 없음** (필드 정의만 존재) | 실제 필드/단위/시스템에 **부합하는 현실적 더미값**으로 재구성 |

> ⚠️ 문서는 "데이터 명세서(스키마)"이지 "실측 데이터셋"이 아니다. 따라서 숫자는 실제 시스템·필드·단위에 맞게 일관되게 채우되, 진짜 측정값이 아님을 전제로 한다.

## 2. 마일 ↔ 실제 시스템 매핑 (확정)

현재 `src/data.ts`의 사업자 정보가 **문서와 불일치**(AI가 채운 가상 파트너). 아래로 교정한다.

| 마일 | 실제 시스템 | 사업자(실명) | 핵심 데이터셋(문서) | 현재 data.ts(오류) |
|---|---|---|---|---|
| **Energy** | 신재생에너지 가상거래 서비스 | **케빈랩** | 태양광(단상/삼상) 집계(생산), AI예측발전량, 상계/정산 | 한국전력·서부발전·에이치에너지 |
| **Mobility** | 친환경 배달문화 밸류체인(BSS) | **우아한형제들** | 전기이륜차 전환 라이더수, BSS 교환, 다회용기 | 와이앤엠월드·쏘카·카카오모빌리티 |
| **Safety** | AIoT 침수홍수 통합관제 / IoT 그린배리어 | **스파이어테크놀로지 / 영진기술** | 우수관로수위·맨홀/지표수위·홍수경보 / 미세먼지·기상 | 그린텍·재난안전연구소 |
| **Data** | 정책플랫폼(70/180종) + 평생학습 | 광명시 | 인구·대기·교통·복지·안전 등 84행 데이터셋 | 국가정보진흥원·한양대 |

## 3. 데이터 소스 카탈로그 (확정값)

### 3.1 에너지 — 케빈랩 신재생 가상거래 (주요 필드)
- 집계(생산): 발전량(Wh), 누적발전량(Wh), PV전압/전류/출력, 현재출력(W), 역률(%), 주파수(Hz)
- AI예측: **AI 예측 발전량(kWh)**
- 상계/정산(월): **소비 전력량 / 발전 전력량 / 상계량 / 수혜 전력량 / 기부 전력량** (kWh)
  - ✅ 메인 에너지 모달의 4개 막대(소비/발전/상계/수혜)와 **이미 일치** → 사업자 연계 문구만 "케빈랩"으로

### 3.2 모빌리티 — 우아한형제들 친환경 배달 밸류체인 (주요 필드)
- 전기이륜차: **사업참여 라이더수(totalRiderCount, 명)**, 전기이륜차 이동거리, 배달건수
- BSS(배터리 스왑): 교환 횟수, 충전 전력량
- 다회용기: 다회용 식당 수, 다회용기 배달 주문 수
  - ✅ 메인 모빌리티 모달 셀들과 거의 일치 → 사업자 "우아한형제들"로

### 3.3 세이프티 — 스파이어테크놀로지(침수홍수) + 영진기술(그린배리어)
- 우수관로수위: 단말종류(1맨홀/2지표면수위계), **맨홀뚜껑 개폐(00/01)**, **맨홀수위레벨(0정상~4심각)**, 수위센서 수위(cm)
- 홍수경보: 홍수구분(1하천범람/2도시침수), 경보구분(1계측/2예측), **경보등급(1관심~4심각)**, 경보상태(Open/Close/Change)
- 그린배리어(영진): 미세먼지/초미세먼지(㎍/m³), 온도/습도/풍향/풍속, 토양수분, 기상청 연계
  - ✅ 메인 세이프티 모달(맨홀수위·지표면수위·경보등급·홍수구분)과 일치 → 사업자/시스템명 보정

### 3.4 데이터 — 정책플랫폼 84행 (DataList/CityDataMap 카탈로그 원천)
대표 데이터셋(실명): 실시간 뉴스(네이버 '광명시'), 대기질 예보통보, 측정소별 실시간 측정정보, 초단기실황(기상), 교통돌발상황(UTIC), 교통 CCTV, 학교/유치원/어린이집, 복지시설, 공영주차장, **전기차충전소**, 하천 수위 관측소(너부대교/오금교/광화교/기아대교), 도시공원, 공공도서관, **주민등록인구/세대 통계**, 전입·전출, 세부사업별 예산·집행 등
- 분류 예시: 인구·생활 / 교통·이동 / 환경·기후 / 안전·재난 / 에너지·건물 / 공공시설
- 좌표계: WGS84 / 기상청격자XY / 좌표없음

> 📌 실제 광명시 인구 ≈ 28만 → 현재 CityDataMap "282,927명"은 현실적이므로 유지 가능.

## 6. 결정 필요 / 데이터 공백 (사용자 확인)

1. **데이터 마일 정체성** — 메인 데이터 모달이 "CCM/VCM 탄소거래·탄소감축"으로 되어 있으나, 문서상 데이터마일은 **정책플랫폼/통합 데이터플랫폼**에 가깝다. 탄소거래(CCM/VCM)의 실제 출처 시스템이 있는지 확인 필요.
2. **수치 시드 범위** — 라이브 값이 없으므로 모든 숫자는 현실적 더미. (합의됨: 정적 시드)
3. **추가 문서 반영 깊이** — `세부데이터_선별`(필드 상세), `기능정의서`, `프로세스정의서`, `아키텍처(정의서/설계서)`는 시스템 동작 명세 → DataDetail 필드/주기 정밀화에 Phase별 부분 활용. 전면 반영 여부 결정.
4. **평생학습** 데이터셋 — 정책플랫폼 외 평생학습 시트 별도 반영 범위.
5. **관리자 운영데이터(공지/자료/계정/팝업)** — 문서에 없음. 현 더미 유지 vs 실명/현실화 수준.
